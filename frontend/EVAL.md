# Retrieval evaluation — SP-10 Arc 1 baseline

This document records the golden-set-based retrieval evaluation for
Redline's clause chat. It is the ship-gate evidence for Arc 1 of
SP-10 (hybrid retrieval foundation) and the reference point every
subsequent arc (metadata boost, reranking, cross-doc search) has to
beat. Numbers below are reproducible from checked-in artifacts — no
live API calls required for the BM25 baseline.

---

## 1. Corpus

Six frozen EU-contract fixtures, captured from the live analysis

`src/eval/fixtures/*.json`:

| Slug               | Jurisdiction | Type                | Clauses | Language of `clause_text`        |
| ------------------ | ------------ | ------------------- | ------- | -------------------------------- |
| `nl-freelance`     | NL           | Freelance services  | 13      | English                          |
| `fr-employment`    | FR           | Employment          | 13      | French                           |
| `de-saas-dpa`      | DE           | SaaS + DPA          | 18      | German                           |
| `es-saas-services` | ES           | SaaS services       | 12      | Spanish                          |

## 2. Golden question set

48 questions (8 per fixture, mix of **3 easy / 3 medium / 2 hard**),
frozen in `src/eval/golden-questions.ts`.

**Provenance.** Authored by Claude Opus 4.7 (`claude-opus-4-7`),
deliberately chosen as a different model family from the in-pipeline
Mistral so the questions don't leak authorship bias into retrieval
numbers. Methodology is in [`src/eval/generate-golden-set.md`](src/eval/generate-golden-set.md).

**Review status.** The set currently carries
`reviewed_by: "claude-opus-4-7"` on every entry — generator-model
self-check only. Baseline numbers in this document are labelled
`pre-human-review` until the project owner's review pass lands.
After human review, both `reviewed_by` and the baseline numbers in
`src/eval/baseline.json` will be refreshed in the same PR.

**Cross-model sanity check (deferred).** 10/48 questions are
scheduled to be re-posed to a third model or human reviewer to catch
intra-cluster agreement. Tracked in the methodology doc.

### Tier definitions

- `easy` — noticeable vocabulary overlap between question and target
  clause. A competent BM25 retriever should find it.
- `medium` — question is conceptually about the clause but uses
  different vocabulary from the clause body (typical for the
  non-English contracts: query is in English, `clause_text` is French
  /German/Spanish/Italian/Polish). Vector search should win here.
- `hard` — answer depends on more than one clause, on a cross-
  reference, or on implicit legal reasoning.

## 3. Metrics

| Metric       | Definition                                                                 |
| ------------ | -------------------------------------------------------------------------- |
| `recall@k`   | Fraction of questions where **any** expected clause index appears in top-k |
| `MRR`        | Mean reciprocal rank of the first hit across the full question set         |

Binary hit semantics: multi-index questions (hard tier) count as a
hit if any of the expected indices is retrieved. Partial-recall is a
retriever limitation we surface in the table, not a separate metric.
If future needs require graded relevance, the `GoldenQuestion` schema
has room to carry it.

## 4. Baseline (BM25 only — keyword retriever)

Reproducible on any checkout with `pnpm vitest run src/eval/harness.test.ts`
(or `EVAL_PRINT=1 pnpm vitest run src/eval/harness.test.ts --reporter=verbose`
to print the table). No network calls.

Committed floor: [`src/eval/baseline.json`](src/eval/baseline.json) —
updated deliberately, never auto-pinned. The CI gate asserts every
slice's metrics are ≥ floor; improvements must re-pin in the same PR
or future regressions hide under an inflated ceiling.

### BM25 — overall + per-tier

| -------------- | -- | -------- | -------- | -------- | ------ |
| overall        | 48 | 0.688    | 0.833    | 0.875    | 0.783  |
| tier: easy     | 18 | 0.889    | 1.000    | 1.000    | 0.944  |
| tier: medium   | 18 | 0.444    | 0.722    | 0.833    | 0.618  |

### Observations (pre-human-review)

- **Easy is essentially solved by keyword** (recall@3 = 1.00). That's
  the tier definition doing its job.
- **Medium is where vector has the most room to earn its keep**
  (recall@1 = 0.44 — BM25 misses the right clause at the top on
  more than half the medium questions). Vocabulary mismatch between
  English query and non-English `clause_text` is the main failure
  mode; the English `plain_english` summary attached to each clause
  is what keeps BM25 from being worse than this.
- **Hard plateaus at 0.75** across @1/@3/@5: BM25 either nails one of
  the multiple expected indices on the first hit, or misses all of
  them — the narrow top-3/top-5 expansion barely helps.
- **`de-saas-dpa` is the worst fixture** (recall@1 = 0.25): 18 clauses
  of dense German with many near-duplicate `data protection` clause
  titles confuse BM25 over-and-above the cross-language penalty.
  Hybrid should gain the most here.
- **`pl-distribution` is a perfect 1.00**: questions happen to use
  enough English vocabulary present in the `plain_english` summary
  for BM25 to hit cleanly even though the clause text is Polish.

## 5. Hybrid retrieval (not yet pinned in CI)

The hybrid retriever (`src/lib/retrieval/hybrid.ts`) fuses BM25 with
cosine similarity on `mistral-embed` vectors via reciprocal rank
fusion. The eval harness wraps it through `makeHybridRetriever` in
`src/eval/retrievers.ts` and feeds in a per-question `queryEmbedding`
map.

Hybrid numbers are **not** yet part of the regression gate because
query embeddings for the golden set are not checked in — doing so
would require committing ~200 KB of 1024-dim vectors and re-freezing
them every time a question is edited. Two options are open and
pending a decision:

1. Commit `golden-query-embeddings.json` alongside the fixtures,
   re-freeze via a dedicated test, and add hybrid to the CI floor.
2. Add a live `harness.live.test.ts` gated on `MISTRAL_API_KEY`, run
   on demand (same contract as `freeze.test.ts`).

Once decided, the hybrid row lands in the baseline table alongside
BM25 with the same per-tier + per-fixture cuts, and the **key deltas
to watch** are:

- `tier: medium` recall@5 — BM25 baseline is 0.833; hybrid target is
  a visible lift (≥ +0.05) without regressing easy.
- `fixture: de-saas-dpa` overall — the worst BM25 case is the
  clearest signal for vector's contribution on cross-language.
- `tier: hard` — expected to stay roughly flat under plain hybrid;
  the real lift for multi-clause questions lands in Arc 2 (metadata
  boost + cross-refs).

## 6. Reproducing these numbers

```bash
cd frontend

# BM25 regression gate (runs in ~1s, no network):
pnpm vitest run src/eval/harness.test.ts

# Print the full report (overall/per-tier/per-fixture) to stdout:
EVAL_PRINT=1 pnpm vitest run src/eval/harness.test.ts --reporter=verbose

# Structural tripwire on the golden set:
pnpm vitest run src/eval/golden-questions.test.ts

# Frozen-fixture schema tripwire:
pnpm vitest run src/eval/fixtures/schema.test.ts

# Re-freeze the fixtures (requires MISTRAL_API_KEY, 10+ min, costs tokens):
FREEZE_FIXTURES=1 MISTRAL_API_KEY=... pnpm vitest run src/eval/fixtures/freeze.test.ts
```

## 7. Change policy

- **BM25 baseline numbers in this doc and `baseline.json`** — update
  together in the same commit whenever the harness, retriever
  tokenisation, or fixtures change.
- **Golden set** — any edit to a question requires a passing run of
  `golden-questions.test.ts` and re-pinning the affected baseline
  slices in the same PR.
- **Hybrid / rerank / metadata / cross-ref retrievers (Arcs 2–3)** —
  each gets its own top-level slot in `baseline.json` (e.g. `hybrid`,
  `hybrid_metadata`, `hybrid_rerank`) so ablations compose cleanly
  without stepping on the BM25 floor.
